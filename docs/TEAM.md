## 二、Git 协作流程

### 2.1 分支策略（GitHub Flow）

```
main（生产分支，保护，仅通过PR合并）
  ├── feature/xsc-api-01（01开发：C端API）
  ├── feature/n8n-workflows-01（01开发：n8n工作流）
  ├── feature/dify-kb-01（01开发：Dify知识库）
  ├── feature/b-admin-02（02开发：B端后台）
  ├── feature/batch-analysis-02（02开发：批量分析）
  ├── feature/openmaic-consult-02（02开发：深度咨询）
  ├── feature/brand-custom-02（02开发：品牌定制）
  └── hotfix/xxx（紧急修复）
```

### 2.2 提交规范

```
格式：<type>(<scope>): <subject>

类型：
  feat: 新功能
  fix: 修复bug
  docs: 文档更新
  style: 代码格式（不影响功能）
  refactor: 重构
  test: 测试
  chore: 构建/工具

示例：
  feat(xsc): 添加今晚行动清单生成逻辑
  feat(n8n): 添加报告推送工作流
  fix(api): 修复OCR低置信度判断bug
  docs(prd): 更新产品需求文档
```

### 2.3 协作流程

```
1. 从 main 拉取最新代码
   git pull origin main

2. 创建功能分支
   git checkout -b feature/xxx-01（或 feature/xxx-02）

3. 开发并本地测试

4. 提交到远程分支
   git push origin feature/xxx-01

5. 创建 Pull Request
   - PR标题：feat(xxx): 功能描述
   - PR描述：做了什么、为什么做、测试通过截图
   - 指派审查人：04

6. 04 审查
   - 代码审查通过 → 合并到 main
   - 有修改意见 → 01/02修改 → 再次审查

7. 合并后删除功能分支
```

### 2.4 审查清单（04使用）

```
□ 代码逻辑正确，无 obvious bug
□ 有充分的错误处理（try-catch、边界判断）
□ 有必要的注释（复杂逻辑必须注释）
□ 命名规范（变量/函数名有意义）
□ 无 console.log 残留（生产代码）
□ 无敏感信息泄露（API key、密码）
□ 数据库变更有 migration
□ API 变更有文档更新
□ 测试用例覆盖新增逻辑
□ 性能无明显问题（无 N+1 查询、无死循环）
```

### 4.3 API 规范

```typescript
// 统一响应格式

  success: boolean;
  data?: T;
  error?: string;
  message?: string;
}

return NextResponse.json({
  success: true,
  data: report

// 错误响应
```
