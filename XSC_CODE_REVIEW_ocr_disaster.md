## 一、架构概览

```
拍照上传 → OCR(百度) → 文本提取 → AI分析(MiniMax) → 报告生成 → 练习生成
     ↑                                                      ↓
     └──────────────── 微信模板消息推送 ───────────────────────┘
```

**链路长度**：6 个节点，任何一环出错就会全链崩盘。

---

## 二、今天 OCR 灾难的根因分析

### 问题现象
- 用户上传六年级英语期末试卷（看图填词：plane/sick/high）
- 系统识别成另一套小学英语题（riding/singing/heavier/September）
- 生成的练习跳跃到初中水平（civilization/定语从句/状语从句）

### 根因 1：OCR 模块识别错误（致命）

**代码位置**：`src/lib/ocr.ts`  
**使用 API**：百度 `doc_analysis`（文档分析）

```typescript
// 关键代码
params.append('language_type', 'CHN_ENG')
params.append('words_type', 'handprint_mix')
params.append('layout_analysis', 'true')
params.append('recg_formula', 'true')
```

**问题**：
- 百度 doc_analysis 面向「文档/试卷」场景，但：**没有题目匹配/题库比对机制**
- 识别出的纯文本直接传给下游 AI，没有任何「这是哪套题」的校验
- 如果图片质量差、角度偏、手写干扰，OCR 会 hallucination（编出一段看起来像题目的文字）

**为什么识别成另一套题？**
- 最可能：百度 OCR 在模糊识别时，从内部题库/训练数据里「联想」出了相似的题目文本
- 次要可能：图片压缩/裁剪导致关键区域丢失，OCR 只能识别到部分文字，然后「脑补」了完整题目

**缺少的关键校验**：
```typescript
// 当前代码没有这些
❌ 识别结果与题库的相似度比对
❌ 题目数量一致性检查（试卷 9 题 vs 识别出 11 题？）
❌ 置信度阈值过滤（低于 80% 的文字应该标记为「不确定」）
❌ 人工复核标记（高不确定性时暂停，等待人工确认）
```

### 根因 2：AI 分析没有「答案校验」机制（严重）

**代码位置**：`src/lib/ai.ts`

```typescript
// 分析流程
const fullPrompt = `${promptTemplate}\n\n【作业内容】\n${text}\n\n要求：...`
const aiText = await callLLM(fullPrompt)
const rawReport = extractJson(aiText)
```

**问题**：
- OCR 识别错了 → 传给 LLM 的 prompt 里就是错的题目
- LLM（MiniMax）基于错误的题目文本做分析 → 自然分析出一堆不存在的知识点
- **没有任何「这题我好像没见过」的告警机制**

**英语学科 prompt 尤其薄弱**：
```typescript
english: `你是一位教育数据分析模型，基于小升初英语知识体系...
请按以下 JSON 格式返回...` // 知识点体系几乎为空！
```

对比数学 prompt：
- 数学：有完整的五大模块、25 个子知识点、难度分级规则
- 英语：只有空壳 JSON 模板，没有知识点体系

**结果**：英语试卷分析全靠 LLM 自由发挥， hallucination 概率极高。

### 根因 3：练习生成没有「难度对标」机制（严重）

**代码位置**：`src/lib/exercises.ts`

```typescript
// 难度分布规则（只基于平均得分率，不看原始试卷难度）
if (avgScore >= 85) {
  difficultyDistribution = '基础题2道 + 提高题2道 + 思维拓展题1道'
} else if (avgScore >= 60) {
  difficultyDistribution = '基础题3道 + 提高题2道 + 思维拓展题1道'
}
```

**问题**：
- 练习难度只由「得分率」决定，**完全不参考原始试卷的难度**
- 今天的情况是：OCR 识别错 → AI 分析错 → 薄弱点全是假的 → 得分率计算错误 → 练习难度跳跃
- **没有「试卷难度锚定」**：如果原始试卷是看图填词，练习就不应该出现定语从句

## 三、关键代码缺陷清单

| # | 缺陷 | 位置 | 严重程度 | 说明 |
|---|------|------|---------|------|
| 1 | OCR 无置信度过滤 | ocr.ts | 🔴 P0 | 低置信度文字应标记，不应直接传给 AI |
| 2 | OCR 无题库比对 | ocr.ts | 🔴 P0 | 识别结果应与已知题库比对，偏差过大时告警 |
| 3 | AI 分析无输入校验 | ai.ts | 🔴 P0 | 题目数量/内容异常时应暂停，不应继续分析 |
| 4 | 英语 prompt 无知识体系 | ai.ts | 🟡 P1 | 英语学科没有知识点模块定义，全靠 LLM 自由发挥 |
| 5 | 练习无难度锚定 | exercises.ts | 🟡 P1 | 练习难度应与原始试卷难度匹配，不应只看得分率 |
| 6 | 无人工复核节点 | 全链路 | 🟡 P1 | 整个流程全自动，没有「不确定时暂停等人确认」的机制 |
| 7 | 批改与 OCR 分离 | correct.ts | 🟢 P2 | correct_edu 用百度批改 API，与 OCR 是两条线，没有交叉验证 |

## 四、修复优先级建议

### Phase 1：止血（24 小时内）

**目标**：让今天的灾难不再发生

1. **OCR 增加置信度检查**
   ```typescript
   // ocr.ts 修改建议
   lines.forEach((l) => {
     if (l.confidence?.average && l.confidence.average < 0.8) {
       l.text = `[低置信度] ${l.text}` // 标记不确定文字
     }
   })
   ```

2. **AI 分析增加「输入异常」检查**
   ```typescript
   // ai.ts 修改建议
   if (text.includes('[低置信度]') || text.split('\n').length < 5) {
     throw new Error('识别质量不足，需要重新拍照或人工确认')
   }
   ```

3. **英语 prompt 补齐知识体系**
   - 至少定义：词汇/语法/阅读/写作/听力 五大模块
   - 每个模块下定义 5-10 个子知识点

### Phase 2：校验（1 周内）

1. **增加「试卷难度锚定」**
   - OCR 识别后，先提取「题目类型分布」（填空/选择/作文等）
   - 练习生成时，题目类型分布应与原始试卷一致
   - 如果原始试卷全是看图填词，练习就不该出现改错题

2. **增加人工复核节点**
   - 不确定时（置信度低/题目数量异常/内容不匹配题库），暂停流程
   - 推送给管理员（你）确认后再继续

3. **OCR 与批改交叉验证**
   - 同一套试卷，同时走 OCR + correct_edu 两条线
   - 比对结果，不一致时标记人工复核

### Phase 3：优化（2 周内）

1. **建立题库索引**
   - 常见试卷录入题库（六年级英语期末、小升初数学等）
   - OCR 识别后，先做模糊匹配，确认是哪套题

2. **练习难度动态校准**
   - 不只是得分率，还要参考：原始试卷难度、学生年级、当地教学大纲

---

## 五、一句话总结

> **今天的灾难不是 AI 分析错了，是 OCR 识别错了，而整个链路没有任何机制拦住错误识别的结果。**

修 OCR 的置信度检查 = 修水龙头。  
修 AI 的输入校验 = 修下水道。  
两者都修，才不会家里发大水。
